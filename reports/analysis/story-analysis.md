## 一、剧情数据结构分析

### 1.1 核心类型定义

剧情系统基于 `StoryNode` 类型构建，位于 `src/types/index.ts`:

```typescript
interface StoryNode {
  id: string;
  type: 'dialogue' | 'choice' | 'battle' | 'reward' | 'narration';
  speaker?: string;           // 对话节点的说话者
  content: string;            // 剧情文本内容
  choices?: StoryChoice[];    // 选择分支
  nextNodeId?: string;        // 下一节点ID
  conditions?: StoryCondition[];  // 触发条件
  effects?: StoryEffect[];    // 效果
  battleId?: string;          // 战斗节点关联副本
}
```

**评价**: 类型设计合理，支持多种节点类型和复杂的条件/效果系统。

### 1.2 章节配置结构

章节配置位于 `src/data/stories/chapters.ts`:

```typescript
interface StoryChapter {
  id: string;
  name: string;
  description: string;
  order: number;
  unlockConditions: StoryCondition[];
  startNodeId: string;
  rewards: ChapterReward[];
}
```

**优点**:
- 支持灵活的解锁条件
- 章节完成有奖励机制
- 顺序控制清晰

### 3.2 分支设计评价

**优点**:
1. 选择影响属性值，有实际游戏效果
2. 通过 flag 标记记录玩家选择，为后续剧情铺垫
3. 部分选择导致不同路线 (如追踪妖狼的三条路)
4. 有 NPC 好友系统雏形 (刘青)

**不足**:
1. 分支最终大多汇聚到同一结局
2. 选择对后续章节影响较小
3. 序章的出身差异在后续章节未能体现
4. 缺少"坏结局"或失败分支

## 六、剧情空缺与问题识别

### 6.1 内容空缺

1. **第四章缺失**: 章节配置存在但剧情节点未实现
2. **后续章节规划不明**: 仅有4章配置，主线延续性存疑
3. **支线剧情缺失**: 目前无任何支线剧情
4. **出身差异未延续**: 三种出身在第二章后无差异化内容

### 6.2 连贯性问题

1. **序章与第二章衔接**:
   - 序章结束于"开始修炼"
   - 第二章开始于"修炼数月"
   - 中间过渡较为生硬

2. **NPC 连续性**:
   - 刘青在第二章结尾消失，后续无交代
   - 王大力被击败后无后续互动
   - 村长/青山村后续剧情潜力未挖掘

3. **标记未使用**:
   - `liu_friend` (刘青好友)
   - `revenge_vow` (复仇誓言)
   - `seeking_truth` (追寻真相)
   - 这些标记在后续剧情中未被利用

### 6.3 系统性问题

1. **缺乏失败分支**: 所有选择都导向成功
2. **战斗结果固定**: 战斗失败对剧情无影响
3. **时间线模糊**: 各章节时间跨度不明确
